# fable-to-opus

> **miss you, fable 🤍**

Opus(또는 지금의 어떤 Claude 모델이든)가 **Fable이 효과적이게 만들었던 작업 습관**을
이어받도록 돕는, 작고 정직한 플레이북입니다 — 검증·완수·조사를 *절차*로 박는 습관들.

마법 스위치가 **아니고**, 강제 설치도 **아닙니다**. 오히려 정반대 —
검증된 습관 몇 가지를 **당신의** 환경에 **맞게 적응**시키도록 돕는 것이 전부입니다.

## ⚠️ 먼저 읽으세요 — 복붙하지 말고, 적응시키세요

이 레포는 **참고 플레이북 + 진단 스킬**이지, 그대로 박는 규칙집이 아닙니다.

전부 `CLAUDE.md`에 복붙하면 십중팔구 **더 나빠집니다**: 지시들이 주의를 두고 경쟁하고,
이미 있는 것과 중복되고, 하네스를 비대하게 만듭니다. 여기서 가장 중요한 단 하나의 원칙:

> **먼저 네 하네스를 분석하라. 빠진 것만, 들어갈 자리에 넣어라.**

아래 어떤 패턴이든 채택하기 전에 **네 기존 환경**부터 확인하세요:

- **네 `CLAUDE.md` / `AGENTS.md`** — 이 습관이 이미 박혀 있나? 중복 금지.
- **네 스킬·플러그인** — 이미 그 관심사를 맡은 스킬이 있나? 경쟁 규칙을 더하지 말고 그걸 강화하라.
- **네 훅(hooks)** — 이미 결정론적으로 강제하고 있나? 산문 규칙은 이미 발동하는 훅보다 약하다.
- **배치 위치** — *모든* 작업에 적용돼야 할 습관은 상시-온 설정에, 산출물 도메인 규칙은
  해당 스킬·가드에, 강제 정지선은 훅에 둔다.

이 레포의 `/fable-to-opus` 스킬이 바로 이 진단을 **대신 해줍니다**: 네 CLAUDE.md를 읽고,
스킬 목록을 보고, 빠진 습관 각각이 *네* 하네스의 *어디에* 들어가야 하는지를 제안합니다 —
규칙을 위에 쏟아붓는 게 아니라. 이게 의도된 사용법입니다.

> 이 플레이북 자체가 처음 채택된 방식이 그랬습니다 — 6개 패턴 중 *정말로 빠져 있던* 2개만
> 기존 하네스에 이식했고, 나머지는 이미 커버돼 있어서 더 넣으면 잡음이 됐을 뿐입니다.

## 왜 만들었나

같은 프로젝트를 나란히 작업해보면, Fable은 일을 *끝내는* 느낌이었습니다 — 촘촘한 실행 리듬,
다단계 편집을 끝까지 밀어붙이는 추진력, 구간 끝마다 깔끔한 요약 보고. Fable을 쓸 수 없게 되어
작업이 Opus로 넘어가면서 질문은 단순했습니다: **그 습관들 중 무엇이 프롬프트로 실제 이식
가능하고, 무엇은 애초에 모델 얘기가 아니었나?**

그래서 두 모델이 *같은* 프로젝트에서 남긴 트랜스크립트를 파싱해 비교했습니다
([docs/ANALYSIS.md](./docs/ANALYSIS.md)). 결과는 냉정하면서도 유용합니다: **어떤 습관은 깨끗이
이식되고, 여러 차이는 그저 역할·국면 교란이었으며, 일부는 모델 내재라 프롬프트로 전혀 안
움직입니다.** 이 레포는 *실제로 이식되는 부분만* 담고, 그렇다고 정직하게 밝힙니다.

하네스는 모델의 천장을 못 올립니다. 검증과 완수를 절차로 바꿔, 모델이 *자기* 천장까지 더
안정적으로 도달하게 할 뿐입니다. 천장 자체가 병목이면, 어떤 프롬프트도 못 고칩니다.

## 6대 이식 가능 패턴 (검증됨)

각 패턴은 [`packs/`](./packs/)에 짧은 팩으로 있습니다. 작업에 맞는 **가장 작은 것**만 적용하세요.

| # | 패턴 | 한 줄 |
|---|------|-------|
| 0 | [적응 우선](./packs/00-adapt-first.md) | 네 하네스를 진단하고, 빠진 것만 들어갈 자리에. **다른 모든 것보다 먼저 읽기.** |
| 1 | [렌더 검증](./packs/01-render-verification.md) | 사람이 보는 산출물(문서·슬라이드·HTML·SVG·차트·UI)은 **렌더된 결과를 직접 봐야** 완료 — 스키마/lint/빌드 통과는 무결성이지 화면 정합이 아니다. |
| 2 | [가설 기반 조사](./packs/02-investigation-hypotheses.md) | 첫 원인에 못 박지 마라. 가설 2개+ 세워 각각 데이터로 기각하고 살아남은 것을 원인으로. |
| 3 | [위임 가드](./packs/03-delegation-guards.md) | 모든 서브에이전트·병렬 위임에 반환 스키마·소스 우선순위·중복배제·날조금지 조항을 박는다. |
| 4 | [입력 충실도](./packs/04-input-fidelity.md) | 스캔·OCR·스크린샷에서 흐릿한 핵심 수치·날짜·식별자는 **고해상도 재추출 후** 신뢰한다. |
| 5 | [비가역 작업 전 확인](./packs/05-irreversible-confirm.md) | 크거나 비가역인 작업(대량 생성·팬아웃·일괄 편집·파괴적 작업) 전에 핵심 가정을 먼저 확인. |
| 6 | [리듬과 구간 보고](./packs/06-rhythm-and-briefing.md) | 명백한 다단계 편집은 한 호흡으로 조용히 밀고, 끝에 **1회** 종합 보고, 선택지로 끝날 땐 권장 디폴트를 곁들여 — 검증 투명성을 죽이지 않으면서. |

## 이식되지 *않는* 것 (정직한 한계)

분석 가치의 절반은 여기 있습니다. 쫓지 마세요:

- **도구 횟수 = 생산성.** "편집을 더 했다 / 서브에이전트를 더 띄웠다"는 대부분 **역할·국면
  교란**이었습니다 — 같은 프로젝트의 다른 구간을 맡았을 뿐. 도구 횟수를 모델 우열로 읽으면 오독.
- **"그냥 빨리 시작하고 덜 물어라."** 확인을 건너뛰는 게 결단력처럼 보였지만, 그건 *어느 국면*을
  맡았느냐와 묶여 있었습니다. 비가역 작업에 밀어붙이면 틀린 산출을 대량생산합니다. 패턴 5 참고.
- **"더 생각해라."** 내부 추론 빈도는 모델 측 특성이라 프롬프트로 안정적으로 못 움직입니다 —
  데이터상 더 많이 추론한 쪽이 자기번복을 막지도 못했습니다.
- **모델의 천장.** 열린 창의적 깊이와 자기주도 발견은 절차가 아니라 능력입니다. 하네스는 오버헤드를
  우회시킬 뿐, 실력을 더하지 않습니다.

## 어떻게 도출했나 (근거)

- **출처:** Fable과 Opus가 *같은* 작업을 나눠 처리한 두 프로젝트 트랜스크립트 약 31MB —
  비슷한 하위작업에 대한 자연 A/B.
- **방법:** 정량 파싱(도구 분포·실행 리듬·검증율·추론 빈도) + 다중 에이전트 분석으로 모든
  차이 주장을 **적대적으로 검증**하고 **역할·국면 교란을 감산**.
- **핵심 발견(가장 견고, 교란 낮음):** 결정적 격차는 *검증의 종착점*이었습니다 — 코드 레벨 검증
  (스키마·ZIP·lint·빌드) 통과를 "완료"로 본 반면, 신뢰 가능한 길은 **실제 렌더 결과를 눈으로
  보는 것**이었습니다. 그 습관이 패턴 1입니다.

전체 보고: [docs/ANALYSIS.md](./docs/ANALYSIS.md).

## 설치

Claude Code 플러그인으로 (권장 — `/fable-to-opus` 진단 스킬이 딸려옵니다):

```bash
/plugin marketplace add silobust-hash/fable-to-opus
/plugin install fable-to-opus@fable-to-opus
```

아니면 그냥 팩을 읽으세요. 전부 평범한 마크다운이라 설치가 필요 없습니다.

### 선택, 옵트인: CLAUDE.md에 포인터 블록 심기

```bash
bash setup/setup.sh local    # 이 프로젝트만 (권장)
bash setup/setup.sh global   # 모든 프로젝트
```

전체 규칙이 아니라 **짧은 포인터**만 심습니다 — 어떤 패턴을 빌리기 전에 적응 우선 진단을
먼저 돌리라고 모델에게 알려주는 한 문단. 언제든 제거:

```bash
bash setup/uninstall.sh
```

규칙을 강제 주입하는 상시-온 훅은 **의도적으로 없습니다** — 그건 적응 우선 원칙과
모순되니까요. 무엇을 하네스에 들일지는 당신이 통제합니다.

것이 어디에 들어가야 하는지 **승인받을 변경셋**으로 제안합니다 — 덮어쓰기가 아니라.
